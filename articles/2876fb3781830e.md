# はじめに

Facebook Login が OpenID Connect (OIDC) に対応？[^1]しているっぽいことに気づいて色々手元で試したので備忘録として残します。
意外と試している人が少なそうに感じたのでどなたかの役に立てば幸いです。なお、本記事の執筆時点ではドキュメント[^1]に `Be aware that this functionality is still in testing.` と記載がある点について十分にご留意ください。

# 事前準備

まだ、Facbook アプリを作成していない場合は、[Create an App with Meta](https://developers.facebook.com/docs/development/create-an-app) からアプリを作成します。
アプリが作成できている場合、以下のようにコンソール上で App ID や App secret を確認することができます。これらの値は後述の認可リクエストやトークンリクエストなどを組み立てる際などに必要となります。

# Authorization Request

ドキュメント[^1]に記載されているサンプルの Authorization Request は以下の通りです。各パラメーターの説明についてもドキュメント[^1]上に記載されているためここでは省略を省きます。基本的には OIDC 仕様[^2]に定義されているパラメーターであり、PKCE 仕様[^3]もサポートしているようです。なお、`client_id` としては先ほど確認した App ID を指定します。

```bash
https://www.facebook.com/v11.0/dialog/oauth?
  client_id={app-id} // Replace with your application’s ID
  &scope=openid
  &response_type=code
  &redirect_uri={"https://www.domain.com/login"}  // Replace with your Redirect URI      
  &state={"state123abc"} // Replace with your State param
  &code_challenge={"E91k-123k123-115X"} // Replace with your generated code_challenge
  &code_challenge_method=S256 // Replace with the method used to generate the code_challenge
  $nonce={"123"} // Replace with a randomly generated nonce value
```

その他に指定可能な `scope` などのパラメーターについては、Manually Build a Login Flow[^3] から確認することができます。
なお、Authorization Endpoint のパスが `/v11.0/dialog/oauth` というあまり見ない形式である点については注意が必要かもしれません。

# Token Request

ドキュメント[^1]に記載されているサンプルの Token Request は以下の通りです。
特筆すべきは Token Endpoint に対して POST ではなく GET でリクエストする点です。各パラメーターについては OIDC 仕様[^2]中などにも定義されている値です。なお、`client_secret` もオプショナルで指定することができ、client_secret を指定しな場合は `code_verifier` を指定する必要があるという動作のようです。

```bash
GET https://graph.facebook.com/v11.0/oauth/access_token?
  client_id={app-id} // Replace with your application’s ID
  &redirect_uri={"https://www.domain.com/login"}  // Must match the Redirect URI stated in the Authorization Request
  &code_verifier={code-verifier} // Replace with generated code verifier
  &code={authorization-code} // Replace with Authorization Code 
```

Token Endpoint についても比較的珍しい形式のパスに見えるので注意が必要かもしれません。

# 動作確認

以下のスクリプトを使って動作を確認します。

```bash
#!/bin/bash

AUTHZ_ENDPOINT="https://www.facebook.com/v11.0/dialog/oauth"
TOKEN_ENDPOINT="https://graph.facebook.com/v11.0/oauth/access_token"
CLIENT_ID="<App ID>"
CLIENT_SECRET="<App Secret>"
RESPONSE_TYPE="code"
REDIRECT_URI="http://localhost:8000/callback"
SCOPE="openid"
STATE=$(uuidgen)
NONCE=$(uuidgen)
CODE_VERIFIER="dBjftJeZ4CVP-mB92K27uhbUJU1p1r_wW1gFWFOEjXk" # https://datatracker.ietf.org/doc/html/rfc7636#appendix-B
CODE_CHALLENGE="E9Melhoa2OwvFrEMTJguCHaoeK1t8URWbuGJSstw-cM" # https://datatracker.ietf.org/doc/html/rfc7636#appendix-B
CODE_CHALLENGE_METHOD="S256"

PRAM="client_id=${CLIENT_ID}&response_type=${RESPONSE_TYPE}&redirect_uri=${REDIRECT_URI}&state=${STATE}&scope=${SCOPE}&nonce=${NONCE}&code_challenge=${CODE_CHALLENGE}&code_challenge_method=${CODE_CHALLENGE_METHOD}"
open "${AUTHZ_ENDPOINT}?${PRAM}"

RESP=$(cat << EOS
HTTP/1.1 200
Server: netcat!
Content-Type: text/html; charset=UTF-8
Check your command line
EOS
)

AUTHZ_CODE=$(echo "${RESP}" | nc -l 8000 | grep GET | cut -d ' ' -f 2 | cut -d '=' -f 2 | cut -d '&' -f 1)
echo ${AUTHZ_CODE}

TOKEN_RESP=$(curl -s -X "GET" "${TOKEN_ENDPOINT}?client_id=${CLIENT_ID}&redirect_uri=${REDIRECT_URI}&code_verifier=${CODE_VERIFIER}&code=${AUTHZ_CODE}")
echo ${TOKEN_RESP}
echo ${TOKEN_RESP} | jq .
```

結果として以下のようなレスポンスが確認できました。

```json
{
  "access_token": "EAAO...",
  "token_type": "bearer",
  "expires_in": 5183654,
  "id_token": "eyJh..."
}
```

なお、Token Request において `code_verifier` および `client_secret` を含めなかった場合、以下のレスポンスになりました。

```json
{
  "error": {
    "message": "No code_verifier specified when a code challenge is provided in\n authorization request",
    "type": "OAuthException",
    "code": 1,
    "fbtrace_id": "xxxx"
  }
}
```
