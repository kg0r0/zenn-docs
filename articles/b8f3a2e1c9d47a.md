本投稿は、OAuth 2.0 Security Best Current Practice (RFC 9700) を拡張する新しいドラフト "Updates to OAuth 2.0 Security Best Current Practice" の差分を見守る会です。今後見守り続けるかは不明ですが、新規の手法やトレンドなどを追っていくうえでの一つのアプローチになるかもと思い、とりあえず試してみます。

なお、今回対象とするのは [draft 00](https://datatracker.ietf.org/doc/draft-ietf-oauth-security-topics-update/00/) であり、比較対象となる過去のドラフトは存在しません。そのため今回は差分ではなく、この仕様がどのような位置付けで、どういった内容を扱おうとしているのかを確認していく構成としています。

## 仕様の位置付け

[draft-ietf-oauth-security-topics-update-00](https://datatracker.ietf.org/doc/html/draft-ietf-oauth-security-topics-update-00) の Abstract には以下のように記載されています。

> This document updates the set of best current security practices for OAuth 2.0 by extending the security advice given in RFC 6749, RFC 6750, and RFC 9700, to cover new threats that have been discovered since the former documents have been published.

つまり、本仕様は既存の以下の RFC で示されているセキュリティ上の推奨事項を「置き換える」のではなく、その後に発見された新しい脅威をカバーするために「拡張」することを目的としているようです。

- RFC 6749 (The OAuth 2.0 Authorization Framework)
- RFC 6750 (Bearer Token Usage)
- RFC 9700 (Best Current Practice for OAuth 2.0 Security, いわゆる Security BCP)

また、RFC 9700 (Security BCP) と併読することを前提としており、RFC 9700 で既に扱われている攻撃については、追加の助言や明確化が必要な場合に限り言及される、というスタンスのようです。

本仕様の中心となるのは "2. Attacks and Mitigations" のセクションで、新たに扱われる 3 つの攻撃が並んでいます。

## 2.1. Audience Injection Attacks

Client 認証で用いられるアサーション (例えば `private_key_jwt` などの JWT 形式の Client 認証、[RFC 7521](https://datatracker.ietf.org/doc/html/rfc7521) / [RFC 7523](https://datatracker.ietf.org/doc/html/rfc7523) で定義される Assertion Framework 系) の "audience" の扱いに関する攻撃です。

### 2.1.1. Attack Description

Client が Authorization Server (以下 AS) の Token Endpoint に対して発行するアサーションにおいて、`aud` クレームに Token Endpoint の URL を指定するプラクティスが一般化していますが、この値だけでは "どの AS 向けのアサーションなのか" を一意に特定できないケースがある、という点が起点になります。

悪意ある AS (attacker-controlled AS) が Client からアサーションを含む正規のリクエストを受け取り、そのアサーションを正規の AS の同じ (もしくは類似の) エンドポイントに対して再送することで、正規の AS では Client として振る舞えてしまう、というシナリオが具体的に記述されています。

影響対象は Token Endpoint だけでなく、Client 認証を要求するあらゆるエンドポイントに及ぶとされています。以下のような拡張仕様のエンドポイントが列挙されています。

- Pushed Authorization Requests (RFC 9126)
- Token Revocation (RFC 7009)
- CIBA (Client Initiated Backchannel Authentication)
- Device Authorization Grant (RFC 8628)

また、影響を受ける Client 認証方式として、`private_key_jwt` や `client_secret_jwt` に代表される、Client が JWT 等のアサーションを生成して送信するタイプの認証方式が挙げられています。

#### 2.1.1.1. Core Attack Steps

draft 00 では、当該攻撃のコアとなるステップが具体的な例と共に説明されています。登場する AS は以下の 2 つです。

- **H-AS (Honest AS)**: 正規の AS。Issuer は `https://honest.com` 、Token Endpoint は `https://honest.com/token` として自身のメタデータ (RFC 8414 / OpenID Connect Discovery) を公開している。
- **A-AS (Attacker-controlled AS)**: 攻撃者が管理する AS。Issuer は `https://attacker.com` として自身を宣伝しつつ、**Token Endpoint として H-AS の `https://honest.com/token` を提示している** (実際には自身が制御していないエンドポイントをメタデータで指し示している)。

この状態で、攻撃のコアとなるステップは以下のように進みます。

1. Client が H-AS に対して登録され、Client ID として `cid` が割り当てられる。
2. Client が A-AS に対しても登録され、同じ Client ID `cid` が割り当てられる (Client ID は秘密情報ではなく、RFC 6749 の Section 2.2 でもその旨が述べられているため、同じ値が使われるケースは十分あり得る)。
3. Client が A-AS に対して Client 認証を行う際、以下のようなクレームを持つ署名付き JWT (アサーション) を生成する。
   - `iss`: `cid`
   - `sub`: `cid`
   - `aud`: `https://honest.com/token` (A-AS のメタデータに従ってセットした結果、実質的には H-AS の Token Endpoint を指すことになる)
4. A-AS は受け取ったアサーションを、そのまま H-AS の `https://honest.com/token` に対して再送 (replay) する。
5. Client は H-AS と A-AS で同じ Client ID・同じ鍵ペアを利用しているため、アサーションは H-AS 側でも有効な Client 認証として通ってしまい、攻撃者は Client として振る舞える。

また、draft 00 では、"アサーションの送信先が Token Endpoint でなくとも、`aud` として Token Endpoint を指定することが多くの標準で許容ないし推奨されている" 点にも触れており、以下のような仕様群がその例として挙げられています。

- RFC 7521 / RFC 7522 / RFC 7523
- RFC 9126 (Pushed Authorization Requests)
- OpenID Connect Core / CIBA
- IANA の OAuth Token Endpoint Authentication Methods レジストリを参照する各仕様

この "送信先エンドポイントに関わらず `aud` に Token Endpoint の URL を指定するプラクティス" が、攻撃を成立させやすくしている背景として整理されています。

### 2.1.2. Countermeasures

対策として以下の 2 つが提示されています。

- **Authorization Server Issuer Identifier**
  - `aud` クレームには、AS の Issuer Identifier (RFC 8414 / OpenID Connect Discovery で定義されているもの) を **唯一の値** として指定する。
- **Exact Target Endpoint URI**
  - `aud` クレームには、送信対象の **正確なエンドポイント URI** を **唯一の値** として指定する。

いずれも "唯一の値として指定する" ことが重要で、複数の URL を並べたり、汎用的な文字列を含めたりすると、再送 (replay) の余地を残してしまう、という整理になっています。

## 2.2. Cross-tool OAuth Account Takeover

こちらは RFC 9700 の Section 4.4 で扱われている、いわゆる Mix-Up Attack を一般化・拡張したような位置付けの攻撃です。

### 2.2.1. Attack Description

昨今、いわゆる Agentic AI や OaaS (OAuth as a Service) 系のプラットフォームなど、ひとつの Client が複数の "ツール" や "プロバイダ"、"テナント" にまたがって OAuth フローを実行する構成が増えてきています。draft 00 では、これらの構成において、Cross-tool OAuth Account Takeover (COAT) と呼ぶ攻撃が成立し得るとしています。
過去に議論されていた "Cross-app OAuth Account Takeover" 系の研究や Mix-Up Attack を、より広い文脈 (複数ツール・複数プロバイダ・複数テナント) で一般化した攻撃と捉えることができるようです。

#### Preconditions

draft 00 では、この攻撃が成立するための前提条件 (Preconditions) が以下の 4 点にまとめられています。

1. Client が、Implicit Grant または Authorization Code Grant を用いて、**複数の OAuth Connection Context** を扱えるようになっていること。具体的には、少なくとも 1 つの正規の組み合わせ (H-Tool / H-AuthProvider / H-AS) と、1 つの攻撃者が制御する組み合わせ (A-Tool / A-AuthProvider / A-AS) が存在する。
2. Client が、ユーザーが選択した Connection Context を、そのユーザーのブラウザに紐づいた **セッションに保存** している。
3. Client の redirect_uri が、"auth provider / tool / tenant" といった Connection Context を構成する変数すべてを織り込んだ形式には **なっていない** 。
4. AS 側が redirect_uri に対して **厳密な完全一致マッチング** を行っている (そうでない場合、Cross Social-Network Request Forgery のような別の攻撃が成立してしまう)。

#### Attack on the authorization code grant

Authorization Code Grant を対象とした攻撃シナリオも具体的に記述されています。

前提として、Client は H-AS (`https://honest.as.example`, Client ID `7ZGZldHQ`) と A-AS (`https://attacker.example`, Client ID `666RVZJTA`) の両方に登録されており、redirect_uri はそれぞれ別のパス (`/honest-cb` / `/attack-cb`) が割り当てられているものとします。

攻撃は以下のステップで進みます。

1. **被害者が攻撃者のツール経由でフローを開始する**: 被害者が、A-Tool (例えば Agentic AI サービス上の何らかのツール実行) を通じて A-AS で始まるフローを選択する。
2. **Client が Connection Context を保存して A-AS にリダイレクトする**: Client は選択された Connection Context をユーザーのセッションに保存し、A-AS の Authorization Endpoint へ、A-AS の Client ID・`state`・攻撃者側の redirect_uri とともにブラウザをリダイレクトする。
3. **A-AS が H-AS へ差し替えてリダイレクトする**: A-AS は受け取ったリクエストの `state` を保ちつつ、Client ID を H-AS のもの (`7ZGZldHQ`) に、redirect_uri を H-AS 側の正規パス (`/honest-cb`) に差し替えたうえで、ブラウザを H-AS の Authorization Endpoint へリダイレクトする。
4. **H-AS が (再同意なしに) 認可コードを発行する**: 既に同意済みであるなどの理由により、被害者側で再同意の画面が出ないまま、H-AS が認可コードを発行し、Client の正規コールバック (`/honest-cb`) にリダイレクトする。
5. **Client がコードを A-AS の Token Endpoint に投げてしまう**: Client のセッションには「A-Tool 経由で始まったフロー」という Connection Context が残っており、`state` の検証も通ってしまうため、Client は受け取った認可コードを **A-AS の Token Endpoint** に対して送信してしまう。
6. **攻撃者が H-AS が発行した認可コードを入手する**: 結果として攻撃者は H-AS が発行した認可コードを手に入れる。Public Client であれば直接トークン交換が可能であり、そうでなくとも RFC 9700 の Section 4.5 で述べられている **Authorization Code Injection Attack** に繋げることで、最終的に "被害者のアカウントを、攻撃者側のツール上で操作できる" 状態を作り出せてしまう。

(以下、[draft 02](https://datatracker.ietf.org/doc/draft-ietf-oauth-security-topics-update/02/) の別セクションで用いられているシーケンス図のスタイルに寄せて、AI に生成させたイメージ図です。)

この攻撃の本質は、Client 側で "どの Connection Context (どの AS / どのツール / どのテナント) 向けのレスポンスなのか" を厳密に紐づけていない点にあります。特に OaaS のように単一の Client 設定を複数テナントで共有している構成では、他テナント方向にまで影響が広がり得る点が draft 00 でも強調されています。

#### 派生パターン (Variants)

draft 00 では、Authorization Code Grant への攻撃をベースに、以下の 4 種類の派生パターンが紹介されています。

##### COAT under the OaaS Context

マルチテナントの OaaS (OAuth as a Service) デプロイメントを対象とした派生パターンです。悪意あるテナントは、以下のいずれかの方法で他テナントに対する攻撃を成立させ得るとされています。

- プラットフォーム側で許容されている場合、あらかじめ構築された OAuth Provider (Client 登録も含む) を持つ **共用のツール** を利用する。
- 別テナントが既に利用している正規の Authorization Server を対象とした、独自の OAuth Provider を備えた **カスタムツール** を追加する。

いずれの経路でも、脆弱な OaaS を共有しているテナント間で **クロステナントのアカウント乗っ取り** が発生し得る、というのがこの派生パターンの主張です。

##### Implicit Grant

Authorization Code Grant への攻撃とほぼ同じフローで進行しますが、ステップ 4 で受け取るのが **認可コードではなくアクセストークン** になる点が異なります。
draft 00 では、Client は "A-AS 向けのフローが完了した" と認識しているため、A-AS の UserInfo Endpoint や、A-AS 配下の Resource Server にリクエストを投げた際に、A-AS 側 (つまり攻撃者) にアクセストークンが渡ってしまう、と説明されています。

##### Cross-tool OAuth Request Forgery (CORF)

Client が選択された Connection Context を **セッションではなく redirect_uri に埋め込む** 実装をしている場合に成立する派生パターンです。
この派生パターンの目的は、通常の COAT のように "被害者の認可コードやアクセストークンを盗む" ことではなく、**"攻撃者側の H-AS 向け認可コードやアクセストークンを、被害者の Client に使わせる"** ことにあります。結果として、被害者のプラットフォーム上のアイデンティティが、攻撃者側のツールアカウントに紐づけられた OAuth 接続が確立されてしまいます。
過去の "Cross-app OAuth Request Forgery" や "Naïve RP Session Integrity Attack" を、対象となる Connection Context を単なる AS の識別を超えて拡張したものと位置付けられています。

##### OpenID Connect

COAT のいくつかの派生パターンは、OpenID Connect にも適用できるとされています。
具体的には、OpenID Connect Discovery の仕組みを悪用したり、アクセストークンや ID Token を再送することで、Mix-Up Attack と同種の攻撃が成立し得る、と説明されています。詳細については、draft 00 内で参照されている先行研究に委ねられています。

### 2.2.2. Countermeasures

draft 00 の Countermeasures には、対策の柱と、それを RFC 9700 由来の既存 Mix-Up 対策で置き換える場合の条件が記載されています。

#### 主となる対策

まず、Client 側に対して以下の要件が示されています。

- **OAuth Provider の共有禁止**: Client は、**オーナーが異なるツールやテナント間で、Client 登録 (Client Credentials 発行済み) が完了した OAuth Provider を共有してはならない (MUST NOT)**。
- **Connection Context 識別子の構築**: Client は、自身がサポートする OAuth Connection Context を構成する全ての変数を用いて、一意な **Connection Context 識別子** を組み立てる。この識別子には **ツール識別子** を必ず含めなければならない。
  - 1 つのツールに対して複数の OAuth Provider の利用を許容する Client (RFC 9700 の Section 4.4 で議論されているように、AS 単位で侵害される可能性がある構成) では、**OAuth Provider 識別子** も含める必要がある。
  - クロステナントで動く Client では、ツール識別子がグローバルに一意でない場合、**テナント識別子** も含める必要がある。
- **Connection Context 単位の redirect_uri と検証**: 明示的に別途規定されない限り、Client は以下を行う。
  1. Connection Context 識別子を組み込んだ、**Connection Context ごとに異なる redirect_uri** を発行する。
  2. 認可リクエストを開始する際に、その Connection Context 識別子を **ユーザーのセッションに保存** する。
  3. Redirection Endpoint で認可レスポンスを受け取った際に、**redirect_uri に含まれる Connection Context 識別子** と、セッションに保存しておいた Connection Context 識別子が **一致することを確認** し、不一致であればフローを中断する。

#### RFC 9700 の Mix-Up 対策による代替

上記の対策の代替として、**RFC 9700 の Section 4.4 に既に定義されている Mix-Up 対策** を採用することも認められています。ただしその場合、以下の 4 つの条件をすべて満たす必要があります。

1. Client が **Implicit Grant のサポートを完全に廃止** している。
2. OAuth Provider が AS を、個別のエンドポイント単位ではなく、それらを代表する **抽象的な Issuer Identifier** で指定している。
3. その Issuer Identifier を、Connection Context 識別子の代わりに用いるか、または RFC 9207 に従って別途返却する。
4. Issuer Identifier から実際の AS エンドポイントを引き当てるための **ランタイム解決** (例: RFC 8414 の Authorization Server Metadata による Discovery) を併用する。

draft 00 では、注意点として、**単に AS メタデータの Discovery を「個別エンドポイント定義済みの OAuth Provider を埋めるためだけに使う」形で導入し、Connection Context 識別子ベースの防御を持たない Client は依然として脆弱** であることが明記されています。

いずれにしても、Mix-Up Attack と同様に "どの Connection Context (AS / ツール / テナント) からのレスポンスなのか" を Client 側が確実に紐づけられるようにすることがポイントとなるようです。

## 2.3. Cross-user OAuth Session Fixation

3 つ目は、OAuth 版のセッション固定攻撃 (Session Fixation) と言える攻撃です。

draft 00 では、2.2 の COAT で扱われたようなデプロイメント (OAuth 接続が「プラットフォーム上のユーザー識別子」に紐づき、その識別子は通常ユーザーエージェント上のセッションで保持される、という前提) が **実運用では成り立たないケースがある** ことを起点として説明が展開されています。

具体的な例として挙げられているのが **Cross-user-agent な OAuth デプロイメント** です。認証済みのネイティブアプリ (バックエンドが Confidential Client として振る舞う) が、ツール連携用の URL を **外部ブラウザ** で開くようなケースでは、そのブラウザには Client との既存セッションが存在しません。これを回避するために、Client は URL 内にセッション識別子を埋め込んで、当該認可セッションを **固定する** 実装を取ることがあります。この固定機構が、そのままこの攻撃のベクタとなり得ます。

影響としては COAT (2.2) と類似しますが、COAT が Connection Context の混同を突くのに対し、こちらは **「その接続コンテキストに紐づけるべきユーザー」の取り違え** を突く点が違いになります。

また、一見すると RFC 9700 の Section 4.7 で述べられている "state を用いたブラウザ紐付け" (CSRF 対策) の違反にも見えるものの、この攻撃では **同一の被害者が Authorization Request と Token Request の両方を最後まで完了させる** ため、PKCE (RFC 7636) を含む一般的な CSRF 対策では防げないと明記されています。draft 00 では、その被害の深刻度は通常の CSRF より高い、と位置づけられています。

なお、本セクションは Authorization Code Grant を対象としています。Cross-device 系フローにおける派生パターンは、別ドラフト "OAuth 2.0 Cross-Device Flows Security BCP (CDFS)" の Section 4 に委ねられています。

### 2.3.1. Attack Description

攻撃者が Client 上で OAuth フローを開始し、その途中の URL を何らかの手段で被害者に踏ませることで、**攻撃者のアカウントに被害者のリソースが紐づけられてしまう** 攻撃です。逆パターン (被害者のアカウントに攻撃者側のリソースが紐づく) も考えられ、いずれにせよ "誰と誰の紐付けか" が攻撃者にとって都合よく歪められる点が問題となります。

#### Preconditions

前提として、Client は End-User のセッションを保持するものの、ユーザビリティ上の理由から **Redirection Endpoint で End-User の再認証を行わない (もしくは行えない)** ことが挙げられています。

またこの攻撃は、**PKCE (RFC 7636) や一般的な CSRF 対策 (state パラメーター等) では十分には軽減できない** 点も明記されています。PKCE や state はあくまで "認可リクエストとコールバックの整合性" を担保するものであり、"認可コードを受け取った Client 上のセッションと、認可した End-User の対応関係" までは担保しないためです。

#### Example Attack

登場人物は攻撃者と、対象ツールに既に認証済みで攻撃対象のリソースを保有している被害者 (End-User) の 2 人です。攻撃は以下の 5 ステップで進みます。

1. 攻撃者が脆弱な Client からツールに対する OAuth フローを開始し、**認可リクエスト URL** を入手する。この時点で `state` パラメーターには「攻撃者側で固定可能な認可セッション」がエンコードされている。
2. 攻撃者はこの認可リクエスト URL を、メール・チャット・不正なリンクなどで **被害者に配布** する。
3. 被害者が URL を開くと、prior/implicit approvals によって同意画面が省略される場合もあり、そのまま Client に対して自分のリソースへのアクセスを認可してしまう。
4. Redirection Endpoint がその `state` を受け取ると、**Client は "攻撃者の認可セッション" を固定したうえで OAuth 接続を完了させてしまう**。
5. 結果として、**Client 上における攻撃者のアカウントが、被害者のリソースへのアクセス権を獲得** する。

#### 派生パターン (Variants)

Example Attack の派生パターンとして、Client が Authorization Endpoint へリダイレクトする前に、セッションを固定するための **pre-authorization URL** を発行するケースが紹介されています。draft 00 では Non-normative example として以下のようなリクエスト / レスポンスが示されています。

```http
GET /oauth?auth_session_id=6064f11c-f73e-425b-b9b9-4a36088cdb2b HTTP/1.1
Host: client.com
```

```http
HTTP/1.1 303 See Other
Location: https://as.example/authorize?
          response_type=code&client_id=K9dTpWzqL7&state=b1d8f043
          &redirect_uri=https%3A%2F%2Fclient.com%2Fcb
Set-Cookie: auth_session_id=6064f11c-f73e-425b-b9b9-4a36088cdb2b
```

この派生パターンでは、攻撃者は Authorization Endpoint 向けの URL ではなく、この **pre-authorization URL** を被害者に踏ませます。すると Client 側のセッション固定は、Example Attack のステップ 4 ではなくより早い段階で発生することになります。

### 2.3.2. Countermeasures

draft 00 の Countermeasures は、原則として **「ある End-User が開始した OAuth Connection フローは、必ず同じ End-User が完了させなければならない」** という要件を掲げています。

#### もっとも素直な対策

もっとも素直な対策は、**セッションを固定するのではなく End-User を再認証する** ことです。ただし、以下のようなケースではセッション固定という選択肢自体を排除することが難しい、と明記されています。

- ユーザーセッションを保持する主体と、OAuth Client とが **別々のエンティティ** で動いているケース (別 Origin の分離されたサービス、あるいは OAuth Client を外部の OaaS プロバイダに外出ししているケースなど)

こうしたデプロイメントの実在は先行研究 (research.cuhk2 / research.cuhk3 として draft 00 内で参照) を根拠に説明されています。

#### コア要件: 既存セッションとの紐付け

再認証ができない場合の中核的な要件として、Client は **`state` または pre-authorization URL 経由で新たに固定された認可セッションを、OAuth フローを開始したユーザーエージェント上の既存セッションと紐付けなければならない (MUST)**、とされています。この紐付けの検証は **アクセストークンリクエストを行う前** に済ませる必要があります。

具体的な実装は、Redirection Endpoint がどこにいるかで分岐します。

- **既存セッションを持つ Endpoint と Redirection Endpoint が同一 Origin・同一ユーザーエージェント**: Client は、アクセストークンリクエスト前に、新しく固定されたセッションと既存セッションの紐付けを検証しなければならない (MUST)。
- **Redirection Endpoint が別 Origin もしくは別ユーザーエージェント**: 以下のいずれかを取る。
  1. 既存セッションを持つ Endpoint と Redirection Endpoint を、同一 Origin・同一ユーザーエージェント上に **配置する** ように実装を変更する (そのうえで、上の同一 Origin ケースのルールを適用する)。
  2. 現在の Redirection Endpoint から、既存セッションが有効な Origin/ユーザーエージェントに **さらにリダイレクトする** (HTTP `Location` によるリダイレクト、または RFC 8252 の Section 7 のネイティブアプリ向けリダイレクト)。

後者 (更なるリダイレクトを挟むケース) では、そのリダイレクト先が **攻撃者に制御可能であってはならない (MUST NOT)** ことが明記されています。ここが緩いと、この対策自体がオープンリダイレクト (RFC 9700 の Section 4.11 参照) の脆弱性に転化してしまうためです。リダイレクト後は、いずれにせよ **アクセストークンリクエスト前に既存セッションとの紐付けを検証** する必要があります。

まとめると、OAuth フロー完了後に、その接続が本当に "いま Client を触っている人" のものかを Client 側で担保する、という考え方になっています。
