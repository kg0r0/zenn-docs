本投稿は、OAuth 2.0 Security Best Current Practice (RFC 9700) を拡張する新しいドラフト "Updates to OAuth 2.0 Security Best Current Practice" の差分を見守る会です。今後見守り続けるかは不明ですが、新規の手法やトレンドなどを追っていくうえでの一つのアプローチになるかもと思い、とりあえず試してみます。

なお、今回対象とするのは [draft 00](https://datatracker.ietf.org/doc/draft-ietf-oauth-security-topics-update/00/) であり、比較対象となる過去のドラフトは存在しません。そのため今回は差分ではなく、この仕様がどのような位置付けで、どういった内容を扱おうとしているのかを確認していく構成としています。

## 仕様の位置付け

[draft-ietf-oauth-security-topics-update-00](https://datatracker.ietf.org/doc/html/draft-ietf-oauth-security-topics-update-00) の Abstract には以下のように記載されています。

> This document updates the set of best current security practices for OAuth 2.0 by extending the security advice given in RFC 6749, RFC 6750, and RFC 9700, to cover new threats that have been discovered since the former documents have been published.

つまり、本仕様は既存の以下の RFC で示されているセキュリティ上の推奨事項を「置き換える」のではなく、その後に発見された新しい脅威をカバーするために「拡張」することを目的としているようです。

- RFC 6749 (The OAuth 2.0 Authorization Framework)
- RFC 6750 (Bearer Token Usage)
- RFC 9700 (Best Current Practice for OAuth 2.0 Security, いわゆる Security BCP)

また、RFC 9700 (Security BCP) と併読することを前提としており、RFC 9700 で既に扱われている攻撃については、追加の助言や明確化が必要な場合に限り言及される、というスタンスのようです。

## 目次

draft 00 の目次は以下の通りです。

- 1. Introduction
  - 1.1. Structure
  - 1.2. Conventions and Terminology
- 2. Attacks and Mitigations
  - 2.1. Audience Injection Attacks
  - 2.2. Cross-tool OAuth Account Takeover
  - 2.3. Cross-user OAuth Session Fixation
- 3. Security Considerations
- 4. IANA Considerations

本仕様の中心となるのは "2. Attacks and Mitigations" のセクションで、新たに扱われる 3 つの攻撃が並んでいます。

## 2.1. Audience Injection Attacks

Client 認証で用いられるアサーション (例えば `private_key_jwt` などの JWT 形式のクライアント認証、[RFC 7521](https://datatracker.ietf.org/doc/html/rfc7521) / [RFC 7523](https://datatracker.ietf.org/doc/html/rfc7523) で定義される Assertion Framework 系) の "audience" の扱いに関する攻撃です。

### 2.1.1. Attack Description

Client が Authorization Server (以下 AS) のトークンエンドポイントに対して発行するアサーションにおいて、`aud` クレームにトークンエンドポイントの URL を指定するプラクティスが一般化していますが、この値だけでは "どの AS 向けのアサーションなのか" を一意に特定できないケースがある、という点が起点になります。

悪意ある AS (attacker-controlled AS) が Client からアサーションを含む正規のリクエストを受け取り、そのアサーションを正規の AS の同じ (もしくは類似の) エンドポイントに対して再送 (replay) することで、正規の AS では Client として振る舞えてしまう、というシナリオが具体的に記述されています。

影響対象はトークンエンドポイントだけでなく、Client 認証を要求するあらゆるエンドポイントに及ぶとされています。以下のような拡張仕様のエンドポイントが列挙されています。

- Pushed Authorization Requests (RFC 9126)
- Token Revocation (RFC 7009)
- CIBA (Client Initiated Backchannel Authentication)
- Device Authorization Grant (RFC 8628)

また、影響を受けるクライアント認証方式として、`private_key_jwt` や `client_secret_jwt` に代表される、Client が JWT 等のアサーションを生成して送信するタイプの認証方式が挙げられています。

#### 2.1.1.1. Core Attack Steps

draft 00 では、当該攻撃のコアとなるステップが具体的な例と共に説明されています。

登場する AS は以下の 2 つです。

- **H-AS (Honest AS)**: 正規の Authorization Server。Issuer は `https://honest.com` 、Token Endpoint は `https://honest.com/token` として自身のメタデータ (RFC 8414 / OpenID Connect Discovery) を公開している。
- **A-AS (Attacker-controlled AS)**: 攻撃者が管理する Authorization Server。Issuer は `https://attacker.com` として自身を宣伝しつつ、**Token Endpoint として H-AS の `https://honest.com/token` を提示している** (実際には自身が制御していないエンドポイントをメタデータで指し示している)。

この状態で、攻撃のコアとなるステップは以下のように進みます。

1. Client が H-AS に対して登録され、Client ID として `cid` が割り当てられる。
2. Client が A-AS に対しても登録され、同じ Client ID `cid` が割り当てられる (Client ID は秘密情報ではなく、RFC 6749 の 2.2 でもその旨が述べられているため、同じ値が使われるケースは十分あり得る)。
3. Client が A-AS に対してクライアント認証を行う際、以下のようなクレームを持つ署名付き JWT (アサーション) を生成する。
   - `iss`: `cid`
   - `sub`: `cid`
   - `aud`: `https://honest.com/token` (A-AS のメタデータに従ってセットした結果、実質的には H-AS のトークンエンドポイントを指すことになる)
4. A-AS は受け取ったアサーションを、そのまま H-AS の `https://honest.com/token` に対して再送する。
5. Client は H-AS と A-AS で同じ Client ID・同じ鍵ペアを利用しているため、アサーションは H-AS 側でも有効な Client 認証として通ってしまい、攻撃者は Client として振る舞える。

また、draft 00 では、"アサーションの送信先がトークンエンドポイントでなくとも、`aud` としてトークンエンドポイントを指定することが多くの標準で許容ないし推奨されている" 点にも触れており、以下のような仕様群がその例として挙げられています。

- RFC 7521 / RFC 7522 / RFC 7523
- RFC 9126 (Pushed Authorization Requests)
- OpenID Connect Core / CIBA
- IANA の OAuth Token Endpoint Authentication Methods レジストリを参照する各仕様

この "送信先エンドポイントに関わらず `aud` にトークンエンドポイント URL を指定するプラクティス" が、攻撃を成立させやすくしている背景として整理されています。

### 2.1.2. Countermeasures

対策として以下の 2 つが提示されています。

- **Authorization Server Issuer Identifier**
  - `aud` クレームには、AS の Issuer Identifier (RFC 8414 / OpenID Connect Discovery で定義されているもの) を **唯一の値** として指定する。
- **Exact Target Endpoint URI**
  - `aud` クレームには、送信対象の **正確なエンドポイント URI** を **唯一の値** として指定する。

いずれも "唯一の値として指定する" ことが重要で、複数の URL を並べたり、汎用的な文字列を含めたりすると、リプレイの余地を残してしまう、という整理になっています。

## 2.2. Cross-tool OAuth Account Takeover

こちらは RFC 9700 の Section 4.4 で扱われている、いわゆる Mix-Up Attack を一般化・拡張したような位置付けの攻撃です。

### 2.2.1. Attack Description

昨今、いわゆる Agentic AI や OaaS (OAuth as a Service) 系のプラットフォームなど、ひとつの Client が複数の "ツール" や "プロバイダ" 、"テナント" にまたがって OAuth フローを実行する構成が増えてきています。draft 00 では、これらの構成において、Cross-tool OAuth Account Takeover (COAT) と呼ぶ攻撃が成立し得るとしています。

過去に議論されていた "Cross-app OAuth Account Takeover" 系の研究や Mix-Up Attack を、より広い文脈 (複数ツール・複数プロバイダ・複数テナント) で一般化した攻撃と捉えることができそうです。

#### Preconditions

draft 00 では、この攻撃が成立するための前提条件 (Preconditions) が以下の 4 点にまとめられています。

1. Client が、Implicit Grant または Authorization Code Grant を用いて、**複数の OAuth Connection Context** を扱えるようになっていること。具体的には、少なくとも 1 つの正規の組み合わせ (H-Tool / H-AuthProvider / H-AS) と、1 つの攻撃者が制御する組み合わせ (A-Tool / A-AuthProvider / A-AS) が存在する。
2. Client が、ユーザーが選択した Connection Context を、そのユーザーのブラウザに紐づいた **セッションに保存** している。
3. Client の redirect_uri が、"auth provider / tool / tenant" といった Connection Context を構成する変数すべてを織り込んだ形式には **なっていない** 。
4. AS 側が redirect_uri に対して **厳密な完全一致マッチング** を行っている (そうでない場合、Cross Social-Network Request Forgery のような別の攻撃が成立してしまう)。

#### Attack on the authorization code grant

Authorization Code Grant を対象とした攻撃シナリオも具体的に記述されています。

前提として、Client は H-AS (`https://honest.as.example`, Client ID `7ZGZldHQ`) と A-AS (`https://attacker.example`, Client ID `666RVZJTA`) の両方に登録されており、redirect_uri はそれぞれ別のパス (`/honest-cb` / `/attack-cb`) が割り当てられているものとします。

攻撃は以下のステップで進みます。

1. **被害者が攻撃者のツール経由でフローを開始する**: 被害者が、A-Tool (例えば Agentic AI サービス上の何らかのツール実行) を通じて A-AS で始まるフローを選択する。
2. **Client が Context を保存して A-AS にリダイレクトする**: Client は選択された Connection Context をユーザーのセッションに保存し、A-AS の認可エンドポイントへ、A-AS の Client ID・`state`・攻撃者側の redirect_uri とともにブラウザをリダイレクトする。
3. **A-AS が H-AS へ差し替えてリダイレクトする**: A-AS は受け取ったリクエストの `state` を保ちつつ、Client ID を H-AS のもの (`7ZGZldHQ`) に、redirect_uri を H-AS 側の正規パス (`/honest-cb`) に差し替えたうえで、ブラウザを H-AS の認可エンドポイントへリダイレクトする。
4. **H-AS が (再同意なしに) 認可コードを発行する**: 既に同意済みであるなどの理由により、被害者側で再同意の画面が出ないまま、H-AS が認可コードを発行し、Client の正規コールバック (`/honest-cb`) にリダイレクトする。
5. **Client がコードを A-AS のトークンエンドポイントに投げてしまう**: Client のセッションには「A-Tool 経由で始まったフロー」というコンテキストが残っており、`state` の検証も通ってしまうため、Client は受け取った認可コードを **A-AS のトークンエンドポイント** に対して送信してしまう。
6. **攻撃者が H-AS が発行した認可コードを入手する**: 結果として攻撃者は H-AS が発行した認可コードを手に入れる。Public Client であれば直接トークン交換が可能であり、そうでなくとも RFC 9700 の 4.5 で述べられている **Authorization Code Injection Attack** に繋げることで、最終的に "被害者のアカウントを、攻撃者側のツール上で操作できる" 状態を作り出せてしまう。

この攻撃の本質は、Client 側で "どの Connection Context (どの AS / どのツール / どのテナント) 向けのレスポンスなのか" を厳密に紐づけていない点にあります。特に OaaS のように単一の Client 設定を複数テナントで共有している構成では、他テナント方向にまで影響が広がり得る点が draft 00 でも強調されています。

#### 派生パターン (Variants)

draft 00 では、Authorization Code Grant への攻撃をベースに、以下の 4 種類の派生パターンが紹介されています。

##### COAT under the OaaS Context

マルチテナントの OaaS (OAuth as a Service) デプロイメントを対象とした派生パターンです。悪意あるテナントは、以下のいずれかの方法で他テナントに対する攻撃を成立させ得るとされています。

- プラットフォーム側で許容されている場合、あらかじめビルドされた OAuth Provider (クライアント登録も含む) を持つ **共用のツール** を利用する。
- 別テナントが既に利用している正規の Authorization Server を狙って、独自の OAuth Provider を伴う **カスタムツール** を追加する。

いずれの経路でも、脆弱な OaaS を共有しているテナント間で **クロステナントのアカウント乗っ取り** が発生し得る、というのがこの派生パターンの主張です。

##### Implicit Grant Variant

Authorization Code Grant への攻撃とほぼ同じフローで進行しますが、ステップ 4 で受け取るのが **認可コードではなくアクセストークン** になる点が異なります。

draft 00 では、Client は "A-AS 向けのフローが完了した" と認識しているため、A-AS の UserInfo エンドポイントや、A-AS 配下のリソースサーバーにリクエストを投げた際に、A-AS 側 (つまり攻撃者) にアクセストークンが渡ってしまう、と説明されています。

##### Cross-tool OAuth Request Forgery (CORF)

Client が選択された Connection Context を **セッションではなく redirect_uri に埋め込む** 実装をしている場合に成立する派生パターンです。

この派生パターンの目的は、通常の COAT のように "被害者の認可コードやアクセストークンを盗む" ことではなく、**"攻撃者側の H-AS 向け認可コードやアクセストークンを、被害者の Client に使わせる"** ことにあります。結果として、被害者のプラットフォーム上のアイデンティティが、攻撃者側のツールアカウントに紐づけられた OAuth 接続が確立されてしまいます。

過去の "Cross-app OAuth Request Forgery" や "Naïve RP Session Integrity Attack" を、対象となる Connection Context を単なる AS の識別を超えて拡張したものと位置付けられています。

##### OpenID Connect Variant

COAT のいくつかの派生パターンは、OpenID Connect にも適用できるとされています。

具体的には、OpenID Connect Discovery の仕組みを悪用したり、アクセストークンや ID Token をリプレイすることで、Mix-Up Attack と同種の攻撃が成立し得る、と説明されています。詳細については、draft 00 内で参照されている先行研究 (arXiv:1704.08539 の Appendix A や、arXiv:1508.04324v2 の Section 6 "Malicious Endpoints Attacks") に委ねられています。

### 2.2.2. Countermeasures

対策としては以下のような方針が示されています。

- 異なるオーナー配下の "ツール" や "テナント" 間で Client 登録 (Client ID 等) を共有しない
- (ツール, プロバイダ, テナント) の組み合わせを一意に識別する **接続コンテキスト識別子 (connection context identifier)** を構築する
- 上記のコンテキスト単位で **別々の redirect_uri** を利用する
- セッション内で、想定していたコンテキストと実際に返ってきたレスポンスのコンテキストが一致していることを検証する

Mix-Up Attack と同様に "どの AS からのレスポンスなのか" を Client 側が確実に紐づけられるようにすることがポイントとなりそうです。

## 2.3. Cross-user OAuth Session Fixation

3 つ目は、OAuth 版のセッション固定攻撃 (Session Fixation) と言える攻撃です。

### 2.3.1. Attack Description

攻撃者が Client 上で OAuth フローを開始し、途中の URL などを何らかの手段で被害者に踏ませることで、被害者がフローを完了させてしまい、結果として **攻撃者のアカウントに被害者のリソース (もしくはその接続) が紐づけられてしまう** というシナリオです。逆パターン (被害者のアカウントに攻撃者側のリソースが紐づく) も考えられ、いずれにせよ "誰と誰の紐付けか" が攻撃者にとって都合よく歪められることが問題となります。

draft 00 では、この攻撃に対して **PKCE (RFC 7636) や一般的な CSRF 対策 (state パラメーター等) では十分には軽減できない** ことが明記されています。PKCE や state はあくまで "認可リクエストとコールバックの整合性" を担保するものであり、"認可コードを受け取った Client 上のセッションと、認可した End-User の対応関係" までは担保しないためです。

### 2.3.2. Countermeasures

対策として以下のような方針が示されています。

- 可能であれば、フロー完了時に End-User を **再認証** する
- 新しく確立された OAuth 接続を、認可リクエスト開始前から存在していた **既存のユーザーエージェントセッションと結び付ける**
- Redirection Endpoint が Client 本体と異なる Origin にある場合 (Native App など) には、RFC 8252 Section 7 のガイダンスに従い、コロケートするか、更なるリダイレクトを挟むなどして、同一セッションでの継続性を担保する

方針としては、OAuth フロー完了後に、その接続が本当に "いま Client を触っている人" のものかを Client 側で担保する、という考え方になっています。

## おわり

初回ということもあり、差分ではなく仕様全体の位置付けと、扱われている 3 つの攻撃の概要を確認する内容にしました。

個人的な感想としては、いずれの攻撃も個別に見れば以前から議論されていたパターンに見える一方で、Agentic AI や OaaS のような "ひとつの Client が多数の AS・テナント・ユーザーコンテキストを扱う" 構成の広がりを背景に、改めて BCP として明文化しようという流れが感じられました。

次回、draft 01 以降で差分を追っていく予定です。ではまた！
