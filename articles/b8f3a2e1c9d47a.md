本投稿は "Updates to OAuth 2.0 Security Best Current Practice" という仕様の差分を見守る会です。

かつて "[OAuth 2.1の差分を見守る会](https://zenn.dev/ritou/scraps/098107802f952b)" というスクラップおよびその記事群がありましたが、本投稿ではそのフォーマットを参考にしつつ、OAuth 2.0 Security Best Current Practice (RFC 9700) を拡張する新しいドラフト "Updates to OAuth 2.0 Security Best Current Practice" の差分を追っていきます。

なお、本投稿は本仕様の差分を見守る会の第一回目となります。すなわち今回対象とする [draft 00](https://datatracker.ietf.org/doc/draft-ietf-oauth-security-topics-update/00/) は初版であり、比較対象となる過去のドラフトは存在しません。そのため今回は差分ではなく、この仕様がどのような位置付けで、どういった内容を扱おうとしているのかを俯瞰的に確認していく構成としています。

## 仕様の位置付け

[draft-ietf-oauth-security-topics-update-00](https://datatracker.ietf.org/doc/html/draft-ietf-oauth-security-topics-update-00) の Abstract には以下のように記載されています。

> This document updates the set of best current security practices for OAuth 2.0 by extending the security advice given in RFC 6749, RFC 6750, and RFC 9700, to cover new threats that have been discovered since the former documents have been published.

要するに、本仕様は既存の以下の RFC で示されているセキュリティ上の推奨事項を「置き換える」のではなく、その後に発見された新しい脅威をカバーするために「拡張」することを目的としています。

- RFC 6749 (The OAuth 2.0 Authorization Framework)
- RFC 6750 (Bearer Token Usage)
- RFC 9700 (Best Current Practice for OAuth 2.0 Security, いわゆる Security BCP)

Intended Status は **Best Current Practice (BCP 240)** で、Updates として RFC 6749, 6750, 7521, 7522, 7523, 9700 が挙げられています。RFC 9700 (Security BCP) と併読することを前提としており、RFC 9700 で既に扱われている攻撃については、追加の助言や明確化が必要な場合に限り言及される、というスタンスのようです。

現時点での Authors は以下の方々です。

- Tim Würtele (University of Stuttgart, Germany)
- Pedram Hosseyni (University of Stuttgart, Germany)
- Kaixuan Luo (The Chinese University of Hong Kong)
- Adonis Fung (Samsung Research America, USA)

本仕様の中心となるのは "2. Attacks and Mitigations" のセクションで、新たに扱われる 3 つの攻撃が並んでいます。

なお、[Appendix E. Document History](https://datatracker.ietf.org/doc/html/draft-ietf-oauth-security-topics-update-00) の draft 00 に該当するエントリは以下の通りです。

> * Initial version

初版なので特筆すべき変更履歴はありません。以降のセクションでは各攻撃の概要と対策を確認していきます。

## 2.1. Audience Injection Attacks

Client 認証で用いられるアサーション (例えば `private_key_jwt` などの JWT 形式のクライアント認証、RFC 7521 / RFC 7523 で定義される Assertion Framework 系) の "audience" の扱いに関する攻撃です。

### 2.1.1. Attack Description

Client が Authorization Server (以下 AS) のトークンエンドポイントに対して発行するアサーションにおいて、`aud` クレームにトークンエンドポイントの URL を指定するプラクティスが一般化していますが、この値だけでは "どの AS 向けのアサーションなのか" を一意に特定できないケースがある、という点が起点になります。

悪意ある AS (attacker-controlled AS) が Client からアサーションを含む正規のリクエストを受け取り、そのアサーションを正規の AS の同じ (もしくは類似の) エンドポイントに対して再送 (replay) することで、正規の AS では Client として振る舞えてしまう、というシナリオが具体的に記述されています。

影響対象はトークンエンドポイントだけでなく、Client 認証を要求するあらゆるエンドポイントに及ぶとされています。以下のような拡張仕様のエンドポイントが列挙されています。

- Pushed Authorization Requests (RFC 9126)
- Token Revocation (RFC 7009)
- CIBA (Client Initiated Backchannel Authentication)
- Device Authorization Grant (RFC 8628)

また、影響を受けるクライアント認証方式として、`private_key_jwt` や `client_secret_jwt` に代表される、Client が JWT 等のアサーションを生成して送信するタイプの認証方式が挙げられています。

### 2.1.2. Countermeasures

対策として以下の 2 つが提示されています。

- **Authorization Server Issuer Identifier**
  - `aud` クレームには、AS の Issuer Identifier (RFC 8414 / OpenID Connect Discovery で定義されているもの) を **唯一の値** として指定する。
- **Exact Target Endpoint URI**
  - `aud` クレームには、送信対象の **正確なエンドポイント URI** を **唯一の値** として指定する。

いずれも "唯一の値として指定する" ことが重要で、複数の URL を並べたり、汎用的な文字列を含めたりすると、リプレイの余地を残してしまう、という整理になっています。

## 2.2. Cross-tool OAuth Account Takeover

こちらは RFC 9700 の Section 4.4 で扱われている、いわゆる Mix-Up Attack を一般化・拡張したような位置付けの攻撃です。

### 2.2.1. Attack Description

昨今、いわゆる Agentic AI や OaaS (OAuth as a Service) 系のプラットフォームなど、ひとつの Client が複数の "ツール" や "プロバイダ" 、"テナント" にまたがって OAuth フローを実行する構成が増えてきています。draft 00 では、これらの構成において、Cross-tool OAuth Account Takeover (COAT) と呼ぶ攻撃が成立し得るとしています。

過去に議論されていた "Cross-app OAuth Account Takeover" 系の研究や Mix-Up Attack を、より広い文脈 (複数ツール・複数プロバイダ・複数テナント) で一般化した攻撃と捉えると理解しやすいと思われます。

派生パターンとして以下のようなものが挙げられています。

- Implicit Grant を利用した攻撃バリアント
- Cross-tool OAuth Request Forgery (CORF)
- OpenID Connect を用いた場合のバリアント

### 2.2.2. Countermeasures

対策としては以下のような方針が示されています。

- 異なるオーナー配下の "ツール" や "テナント" 間で Client 登録 (Client ID 等) を共有しない
- (ツール, プロバイダ, テナント) の組み合わせを一意に識別する **接続コンテキスト識別子 (connection context identifier)** を構築する
- 上記のコンテキスト単位で **別々の redirect_uri** を利用する
- セッション内で、想定していたコンテキストと実際に返ってきたレスポンスのコンテキストが一致していることを検証する

Mix-Up Attack と同様に "どの AS からのレスポンスなのか" を Client 側が確実に紐づけられるようにすることがポイントとなりそうです。

## 2.3. Cross-user OAuth Session Fixation

3 つ目は、OAuth 版のセッション固定攻撃 (Session Fixation) と言える攻撃です。

### 2.3.1. Attack Description

攻撃者が Client 上で OAuth フローを開始し、途中の URL などを何らかの手段で被害者に踏ませることで、被害者がフローを完了させてしまい、結果として **攻撃者のアカウントに被害者のリソース (もしくはその接続) が紐づけられてしまう** というシナリオです。逆パターン (被害者のアカウントに攻撃者側のリソースが紐づく) も考えられ、いずれにせよ "誰と誰の紐付けか" が攻撃者にとって都合よく歪められることが問題となります。

draft 00 では、この攻撃に対して **PKCE (RFC 7636) や一般的な CSRF 対策 (state パラメーター等) では十分にはミティゲートできない** ことが明記されています。PKCE や state はあくまで "認可リクエストとコールバックの整合性" を担保するものであり、"認可コードを受け取った Client 上のセッションと、認可した End-User の対応関係" までは担保しないためです。

### 2.3.2. Countermeasures

対策として以下のような方針が示されています。

- 可能であれば、フロー完了時に End-User を **再認証** する
- 新しく確立された OAuth 接続を、認可リクエスト開始前から存在していた **既存のユーザーエージェントセッションと結び付ける**
- Redirection Endpoint が Client 本体と異なる Origin にある場合 (Native App など) には、RFC 8252 Section 7 のガイダンスに従い、コロケートするか、更なるリダイレクトを挟むなどして、同一セッションでの継続性を担保する

いわゆる "OAuth 完了後に、その接続が本当に "いま Client を触っている人" のものか" を Client 側で担保する、という考え方になっています。

## おわり

初回ということもあり、差分ではなく仕様全体の位置付けと、扱われている 3 つの攻撃の概要を確認する内容になりました。

個人的な感想としては、いずれの攻撃も "個別に見れば以前から議論されていた素朴なパターンに見える" 一方で、Agentic AI や OaaS のような "ひとつの Client が多数の AS・テナント・ユーザーコンテキストを扱う" 構成の広がりを背景に、改めて BCP として明文化しようという流れが感じられました。

特に 2.1 の Audience Injection Attacks は、`private_key_jwt` を採用しつつも `aud` をトークンエンドポイント URL のみで指定しているような実装は結構ありそうで、Issuer Identifier を採用するか、送信先エンドポイント URI に統一するか、といった選択を意識しておくとよさそうです。

また、2.3 の Cross-user OAuth Session Fixation については "PKCE / state だけでは防げない" という点がハッキリ書かれたのが大きく、Client 実装者としては見落としやすいポイントなので、次以降のドラフトでどこまで具体的なガイダンスに落ちていくのかを見守っていきたいところです。

次回、draft 01 以降で差分を追っていく予定です。ではまた！
