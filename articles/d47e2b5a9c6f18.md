本投稿は、OAuth 2.0 Security Best Current Practice (RFC 9700) を拡張する新しいドラフト "Updates to OAuth 2.0 Security Best Current Practice" の差分を見守る会です。

前回 (draft 01-02) の記事は以下を参照ください。

https://zenn.dev/kg0r0/articles/c8b3a1e2d94f57

## 差分

[差分](https://author-tools.ietf.org/iddiff?url1=draft-ietf-oauth-security-topics-update-02&url2=draft-ietf-oauth-security-topics-update-03&difftype=--html)

[draft 03](https://datatracker.ietf.org/doc/html/draft-ietf-oauth-security-topics-update-03) における主な差分は、[Appendix "Document History"](https://datatracker.ietf.org/doc/html/draft-ietf-oauth-security-topics-update-03#name-document-history) の -03 エントリに記載の以下の通りです。

> * Note that issuing distinct redirection URIs alone does not prevent the COAT attack
> * Add comparison of the COAT countermeasure with issuer-based mix-up countermeasures
> * Remove a COAT variant mention already covered by Shared Consent in Brokered OAuth
> * Clarify the native app backend scenario in Session Fixation and label it as the BFF pattern
> * Reduce non-standard terminology and unify terms used in the Session Fixation section
> * Add sequence diagram to illustrate Session Fixation attack flow
> * Editorial clarifications and fixes to COAT and Session Fixation section wording

念のため、上から見ていきましょう。なお、当方の主観で対応する変更を記載していますが、認識が間違っている可能性は十分あります。もし、認識間違っている箇所があれば [GitHub](https://github.com/kg0r0) から Issue または PR を作成してご指摘いただけますと幸いです。

### Note that issuing distinct redirection URIs alone does not prevent the COAT attack

2.2.1 Attack Description に、**「Connection Context ごとに redirect_uri を分けているだけでは COAT は防げない」** ことを明示する注意書きが追加されました。

draft 02 では Countermeasures 側で "Connection Context 識別子" ベースの対策が提示されていましたが、Attack Description を読んだだけだと "toolkit ごとに違う redirect_uri を使えば OK なのでは?" と誤解しやすい構造でした。draft 03 では攻撃例の直後にこの誤解を先回りする 1 文が挿入され、"別々の redirect_uri を発行するだけ" では不十分であり、Client 側での Connection Context 識別子の紐付け検証が必要である、という点を強調する形になっています。

![](/images/d47e2b5a9c6f18/diff-distinct-redirection-uris.png)

### Add comparison of the COAT countermeasure with issuer-based mix-up countermeasures

2.2.2 Countermeasures に、**COAT 対策 (Connection Context 識別子ベース) と、既存の Mix-Up 対策 (Issuer Identifier ベース) の位置付けを比較する記述** が追加されました。

Issuer Identifier ベースの Mix-Up 対策は、"各 AS が自身を一意に識別するための Issuer Identifier" に依存します。これに対して、draft 03 の COAT 対策で導入された Connection Context 識別子は、Client 側で "個々の AS インスタンス" を一意に識別する仕組みです。両者は目的が近いように見えつつも、識別の起点が **AS 側 (Issuer Identifier)** か **Client 側 (Connection Context 識別子)** かという違いがあり、COAT 対策はその点で Mix-Up 対策とは異なる補完的な位置付けであることが明確化されました。

前回 draft 01-02 の記事でも触れた「Mix-Up 対策で代替可能な 4 条件」との棲み分けが、より読みやすくなった印象です。

![](/images/d47e2b5a9c6f18/diff-comparison-mix-up.png)

### Remove a COAT variant mention already covered by Shared Consent in Brokered OAuth

2.2.1 Variants で従来言及されていた派生パターンのうち、**Section 2.4 (Shared Consent in Brokered OAuth) でカバーされる内容と重複していた記述が削除** されました。

draft 02 で Section 2.4 が新規追加された時点では、"OaaS における同意の使い回し" は COAT (Cross-toolkit OAuth Account Takeover) の派生の 1 つとしても言及されていましたが、Broker 経由の同意共有はむしろ 2.4 の主題そのものです。draft 03 では 2.2 と 2.4 の役割分担を整理し、**Broker まわりの同意共有シナリオは 2.4 に一本化** される形になりました。

なお、2.2.1 Variants 自体は 4 種類 (COAT under the OaaS Context / Implicit Grant / CORF / OpenID Connect) 構成のまま残っており、章の骨格は変わっていません。

![](/images/d47e2b5a9c6f18/diff-removed-variant.png)

### Clarify the native app backend scenario in Session Fixation and label it as the BFF pattern

2.3 Cross-user OAuth Session Fixation で扱っていた **「ネイティブアプリのバックエンドが Confidential Client として振る舞う」** シナリオが、**BFF (Backend-for-Frontend) パターン** として明確にラベリングされました。

draft 03 では以下のような趣旨の記述が追加されています。

> When a native app's backend acts as a confidential OAuth client (commonly referred to as the backend-for-frontend pattern), the OAuth flow crosses user agents, but the established session does not.

要は、"OAuth フローはユーザーエージェントをまたぐが、確立されるセッションはユーザーエージェントをまたがない" という、Cross-user Session Fixation が成立する典型的な構成の名前を、業界で定着している **BFF パターン** という呼称に合わせた、ということです。読み手が業界用語ですぐ像を結べるようになった点で地味に嬉しい改善に見えます。

![](/images/d47e2b5a9c6f18/diff-bff-pattern.png)

### Reduce non-standard terminology and unify terms used in the Session Fixation section

2.3 全体で、**独自用語の整理と、OAuth 標準に近い用語への統一** が行われました。

draft 02 までは "認可セッション" (authorization session) と "ユーザーセッション" (user session) などが局所的に混在していた箇所がありましたが、draft 03 では **"authorization session" / "user session" / "connection context"** といった軸が Session Fixation セクション全体で統一的に使い分けられる形になっています。

各用語の意味そのものは変わっておらず、"どの粒度のセッションを指しているのか" が読み手にとって曖昧になりにくくなった、という理解でよさそうです。

### Add sequence diagram to illustrate Session Fixation attack flow

2.3.1 Attack Description に、**Session Fixation 攻撃のフローを ASCII で示すシーケンス図** が新規追加されました。個人的には draft 03 でもっとも視覚的なインパクトの大きい変更です。

図のカラム構成は以下のようになっています。

- **Attacker** (攻撃者)
- **(Victim) User** (被害者)
- **Client**
- **AS**
- **RS** (Resource Server)

図中で番号付きのステップとしてラベル付けされているイベントは以下のようなものです。

- Attacker が Client 上で OAuth フローを開始する
- `state=attacker_auth_session` を含む認可リクエスト URL を入手する
- 認可リクエスト URL を Out-of-band (フィッシングリンクなど) で被害者に送りつける
- 被害者が当該 URL を訪問する
- 同意画面がスキップされる場合がある (`may be skipped`)
- `code + state=attacker_auth_session` を含む認可レスポンスが返る
- Client が code をトークン交換に使う
- 発行された **トークンが被害者のリソースに向けたものであるにもかかわらず、Client 側では攻撃者のアイデンティティに紐付けられる**
- 攻撃者がその Client 経由で RS にリソースリクエストを投げる
- 結果として、**被害者のリソースに攻撃者が Unauthorized アクセス** する

これまでは文章と 5 ステップの説明のみで示されていた攻撃フローが、視覚的な形で仕様側に取り込まれたことで、初見の読み手にとってもかなり追いやすくなりました。前回の記事で筆者が自作した ASCII 図を貼っていたのが遠い記憶ですが、これで公式の図がある状態になった、というのも感慨深いところです。

![](/images/d47e2b5a9c6f18/diff-session-fixation-seq.png)

### Editorial clarifications and fixes to COAT and Session Fixation section wording

Document History に単体で立ってはいるものの内容は雑多です。COAT と Session Fixation セクションを中心に、細かい表現の修正や書き直しが行われています。個別に確認したい方は冒頭に貼った差分リンクを参照するのが早いと思います。

## その他の変更

Document History に個別項目として立っていない差分としては、以下のようなものが見受けられます。

- **メタデータの更新**: Publication date と Expiration date が更新 (`24 June 2026` → `6 July 2026`、`26 December 2026` → `7 January 2027`)

## おわり

draft 03 は Section 単位で新規追加 / 削除が発生した回ではなく、**"読み手がハマりやすい誤解を先回りする" / "既存対策との位置関係を明示する" / "図を追加してフローを可視化する"** といった、ドキュメントの読みやすさに直結するリファインが中心の回でした。

個人的には、2.3.1 に公式の Session Fixation シーケンス図が入ったのが良い変更だと思います。文章だけで説明するには手数がかかる攻撃なので、この図はプレゼンやレビュー時にもかなり引用しやすくなりそうです。
