本投稿は、OAuth 2.0 Security Best Current Practice (RFC 9700) を拡張する新しいドラフト "Updates to OAuth 2.0 Security Best Current Practice" の差分を見守る会です。

前回 (draft 00-01) の記事は以下を参照ください。

https://zenn.dev/kg0r0/articles/a2c7e9f3d81b46

## 差分

[差分](https://author-tools.ietf.org/iddiff?url1=draft-ietf-oauth-security-topics-update-01&url2=draft-ietf-oauth-security-topics-update-02&difftype=--html)

[draft 02](https://datatracker.ietf.org/doc/html/draft-ietf-oauth-security-topics-update-02) における主な差分は、[Appendix "Document History"](https://datatracker.ietf.org/doc/html/draft-ietf-oauth-security-topics-update-02#name-document-history) の -02 エントリに記載の以下の通りです。

> * Simplify core attack description for Audience Injection attacks by moving "why token EP" discussion to a new section
> * Include concrete examples for how an attacker may misuse client assertions obtained via Audience Injection attacks
> * Add implementation considerations for Audience Injection countermeasures, including recommendations for dealing with ASs that only accept their token endpoint as audience value
> * Add new section on Shared Consent in Brokered OAuth, describing the attack, its preconditions, and countermeasures

念のため、上から見ていきましょう。なお、当方の主観で対応する変更を記載していますが、認識が間違っている可能性は十分あります。もし、認識間違っている箇所があれば [GitHub](https://github.com/kg0r0) から Issue または PR を作成してご指摘いただけますと幸いです。

### Simplify core attack description for Audience Injection attacks by moving "why token EP" discussion to a new section

2.1.1 Attack Description の内部構成が整理されました。draft 01 までは Core Attack Steps に併記されていた "なぜ Token Endpoint を `aud` として指定するプラクティスが広く使われているのか" という背景説明が **別セクション** に切り出されています。

具体的には、2.1.1 配下のサブセクション番号が以下のように付け替えられています。

- `2.1.1.2` **Token Endpoint as Client Assertion Audience** (背景説明を切り出した新規の位置付け)
- `2.1.1.3` Endpoints Requiring Client Authentication (旧 `2.1.1.2` )
- `2.1.1.4` Affected Client Authentication Methods (旧 `2.1.1.3` )

Core Attack Steps 側は、"どうやって攻撃が成立するか" のステップ列に集中し、"なぜ多くの実装が Token Endpoint を `aud` に指定してしまっているのか" の理由付けは別セクションで整理する、という切り分けになりました。

![](/images/c8b3a1e2d94f57/diff-section-renumbering.png)

### Include concrete examples for how an attacker may misuse client assertions obtained via Audience Injection attacks

2.1 Audience Injection Attacks に、**攻撃者が取得した Client Assertion を実際にどう悪用できるか** の具体例が追記されました。

draft 01 までは "H-AS でも Client 認証として通ってしまう" というところで説明が止まっていましたが、draft 02 では以下のような具体的な悪用シナリオが例示される形になっています。

- H-AS 上で **Client なりすまし** をした状態で、`client_credentials` グラントによって **アクセストークンを取得する**
- 認可コードフローを開始し、その **アクセストークンが最終的に攻撃者側に渡る** 形に持ち込む

「Client Assertion がリプレイできる」ことのその先が具体的にイメージしやすくなった形です。

![](/images/c8b3a1e2d94f57/diff-concrete-examples.png)

### Add implementation considerations for Audience Injection countermeasures, including recommendations for dealing with ASs that only accept their token endpoint as audience value

2.1.2 Countermeasures 配下に **`2.1.2.3` Implementation Considerations** という新規サブセクションが追加されました。

draft 01 までの推奨対策は、`aud` に Issuer Identifier もしくは送信対象エンドポイントの URI そのものを **唯一の値** として指定することでした。しかし実運用では、AS が **Token Endpoint の URL のみを `aud` として受け入れる** 実装になっているケースもあり、上位対策をそのまま採用できない場合があります。

`2.1.2.3` は、そうした AS を相手にする Client 向けに、優先順位付きの代替対策を提示するセクションです。優先度順で以下の 3 案が示されています。

1. **AS ごとに異なる鍵素材を利用する**: AS 間で共有された Client 認証用の鍵を無くすことで、リプレイの余地を潰す
2. **AS 横断でグローバルに一意な Client ID を利用する**: OpenID Federation 環境などで難しいケースがある旨も併記
3. **(Client ID, Token Endpoint) の組がグローバルに一意になるようにする**: そのうえで Token Endpoint を唯一の `aud` として使い続ける

いずれも、"AS 側の受け入れ仕様を変えられない" 状況で Client 側だけでできる緩和策としてのグラデーションになっています。上位対策 (Issuer Identifier ベース / エンドポイント URI ベース) を採用できない場合の実装指針、という位置付けです。

![](/images/c8b3a1e2d94f57/diff-implementation-considerations.png)

### Add new section on Shared Consent in Brokered OAuth, describing the attack, its preconditions, and countermeasures

draft 02 で追加された最大のトピックです。**Section 2.4 "Shared Consent in Brokered OAuth"** という新規セクションがまるごと追加され、これに伴い draft 02 で扱う攻撃は 3 つから 4 つに増えました。

Brokered OAuth デプロイメントを対象にした攻撃で、Broker (B) が複数のダウンストリーム Client を代理して単一の Client 登録で上位 AS に接続している構成における、**同意 (consent) の使い回し** の問題を扱っています。

![](/images/c8b3a1e2d94f57/diff-shared-consent-intro.png)

#### 2.4.1 Attack Description

登場するアクターは以下の通りです。

- **Broker (B)**: ダウンストリーム Client と上位 AS の間に立つ仲介者。上位 AS に対しては Client として振る舞う
- **Honest Client (H-Client)**: 正規のダウンストリーム Client
- **Malicious Client (M-Client)**: 攻撃者が用意した悪意あるダウンストリーム Client
- **Upstream AS**: リソースアクセスを管理する上位 AS

主な前提条件は以下の通りです。

- 同一の Broker に、少なくとも 2 つのダウンストリーム Client (Honest / Malicious の両方) が統合されている
- Broker は上位 AS に対して **1 つの Client 登録のみ** を持ち、これをすべてのダウンストリーム Client で使い回している
- 上位 AS 側からは、Broker の Client ID (例: `cid_B@AS`) しか見えず、その裏にいる個々のダウンストリーム Client を区別できない

この構成では、ユーザーが最初に H-Client 経由で認可すると、AS からは "`cid_B@AS` に対して同意した" ように見えます。続いて M-Client が同じ Broker 経由で認可リクエストを投げると、AS はすでに `cid_B@AS` に同意済みと判断し、**追加の同意画面を出さないまま** M-Client に対してもトークンを発行してしまう、というのがこの攻撃の中心的な流れです。

#### 2.4.2 Countermeasures

対策は 2 つの方向性が示され、それぞれのトレードオフも比較されています。

##### 2.4.2.1 Per-Client Registration

ダウンストリーム Client ごとに、**上位 AS に別々の Client 登録** を行う対策です。Broker は認可フローを開始する際、対象のダウンストリーム Client 用の登録を用いて上位 AS に接続します。これにより、上位 AS 側で各ダウンストリーム Client が別 Client として認識されるため、同意もそれぞれに対して個別に管理されるようになります。

![](/images/c8b3a1e2d94f57/diff-per-client-registration.png)

##### 2.4.2.2 Broker-Side Consent Screen

上位 AS への認可リクエストの前に、**Broker 自身が個別の同意画面** をダウンストリーム Client ごとに表示する対策です。過去に別のダウンストリーム Client に対して同意があったとしても、それを新しい Client に対して黙って流用させない、というアプローチになります。

![](/images/c8b3a1e2d94f57/diff-broker-side-consent.png)

##### 2.4.2.3 トレードオフ

2 つの対策のトレードオフは以下のように整理されています。

| 対策 | UX | 実装コスト |
|---|---|---|
| Per-Client Registration | 認可フローあたり同意画面は最大 1 枚 | 各 Client を各 AS で個別に登録する必要あり |
| Broker-Side Consent Screen | 認可フローあたり同意画面が最大 2 枚 (Broker 側 + AS 側) | Broker の Client 登録は 1 つで済み、実装コスト低 |

どちらも Client 側 (Broker 側) だけで完結する対策であり、上位 AS 側のプロトコル変更は不要である旨も明記されています。

## その他の変更

Document History に個別項目として立っていない差分としては、以下のようなものが見受けられます。

- **メタデータの更新**: Publication date と Expiration date が更新 (`2 March 2026` → `24 June 2026`、`3 September 2026` → `26 December 2026`)
- **References の更新**: 2.4 で参照される先行研究の追加、および 2.2 / 2.3 で参照されていた `[research.cuhk]` / `[research.cuhk3]` 系の識別子が `[research.cuhk]` / `[research.cuhk2]` に整理

## おわり

draft 02 のインパクトはやはり **Section 2.4 Shared Consent in Brokered OAuth の追加** に尽きるでしょう。Broker (仲介) 構成における "同意の使い回し" 問題は、既存の COAT (2.2) や Session Fixation (2.3) とは着眼点が異なる問題であり、"1 つの Broker Client 登録で複数のダウンストリームを差配する" というよくある構成の落とし穴として、明文化された意義は大きそうです。

その他、2.1 の実装上の考慮事項が追加された点も、AS 側の受け入れ仕様を変えられない状況にある Client 実装者が考慮されているように見受けられるのは良さそうです。
