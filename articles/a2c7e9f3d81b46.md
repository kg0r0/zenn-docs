本投稿は、OAuth 2.0 Security Best Current Practice (RFC 9700) を拡張する新しいドラフト "Updates to OAuth 2.0 Security Best Current Practice" の差分を見守る会です。

前回 (draft 00) の記事は以下を参照ください。

https://zenn.dev/kg0r0/articles/b8f3a2e1c9d47a

## 差分

[差分](https://author-tools.ietf.org/iddiff?url1=draft-ietf-oauth-security-topics-update-00&url2=draft-ietf-oauth-security-topics-update-01&difftype=--html)

[draft 01](https://datatracker.ietf.org/doc/html/draft-ietf-oauth-security-topics-update-01) における主な差分は、[Appendix "Document History"](https://datatracker.ietf.org/doc/html/draft-ietf-oauth-security-topics-update-01#name-document-history) の -01 エントリに記載の以下の通りです。

> * Clarify that shared redirection URI is not a precondition of COAT
> * Clarify that COAT countermeasure uniquely identifies each configured AS instance
> * Clarify Session Fixation countermeasures and relationship to CSRF and PKCE
> * Use terminology that is less ambiguous and better aligned with standard OAuth language
> * Editorial clarifications and fixes, and reference updates

なお、当方の主観で対応する変更を記載していますが、認識が間違っている可能性は十分あります。もし、認識間違っている箇所があれば [GitHub](https://github.com/kg0r0) から Issue または PR を作成してご指摘いただけますと幸いです。

### 用語の整理 : "tool" → "toolkit"

もっとも見た目のインパクトが大きい変更は、2.2 で用いられていた用語 "tool" が **"toolkit"** に置き換わっていることです。これは Document History の "Use terminology that is less ambiguous and better aligned with standard OAuth language" に対応する変更と思われます。

具体的には以下のような置き換えが行われています。

- **章タイトル**: `Cross-tool OAuth Account Takeover` → `Cross-toolkit OAuth Account Takeover` (略称の "COAT" は据え置き)
- **アクター**: `H-Tool` / `A-Tool` → `H-Toolkit` / `A-Toolkit`
- **派生パターンの名称**: `Cross-tool OAuth Request Forgery (CORF)` → `Cross-toolkit OAuth Request Forgery (CORF)`

また、あわせて "Platform Integrations" が **"Application Integration"** に、対象を表す語彙も "platform" が **"application"** に整理されているようです。

用語 "toolkit" の位置付けについては、2.2 の導入部で以下のように整理されるようになりました。

> Each set of tools, known as a toolkit, is mapped to an OAuth provider configuration

すなわち、"個々のツールの集合" を **"toolkit"** と呼び、それぞれの toolkit が 1 つの OAuth Provider Configuration にマップされる、という関係整理になっています。

なお、2.2 の Variant として登場する OaaS についても、その別名として **"Token Vault"** という表現が追記されています。

### 2.2.1 Preconditions : redirect_uri の前提条件が変更

2.2.1 の Preconditions において、"shared redirection URI is not a precondition of COAT" という Document History の記載通り、**「Client が redirect_uri を共有していること」は COAT の必要条件ではない** ことが明確化されました。

draft 00 の Preconditions は "Client の redirect_uri が Connection Context の変数すべてを織り込んだ形式には**なっていない**" ことを条件として挙げていましたが、draft 01 ではこれをより端的に、以下の要件として整理し直しています。

> the client issues redirection URIs which do not depend on all variables in the connection context (e.g., auth provider, tool, tenant)

つまり "redirect_uri が Connection Context の全変数に依存していない" という条件に整理された形になります。当該条件を満たす場合の例として、toolkit ごとに **異なる** redirect_uri を発行しているケースも該当し得るという整理になり、"共有 redirect_uri" が必須ではなくなりました。

これに合わせて、Attack Description で挙げられていた具体例も、H-Toolkit の redirect_uri が `https://client.com/honest-cb`、A-Toolkit のそれが `https://client.com/attack-cb` と、**別々のパス** に整理される形で更新されています。

### 2.2.1 Variants : OaaS Context のバリアント記述の順序変更

2.2.1 の派生パターン (Variants) のうち、"COAT under the OaaS Context" の記述順が入れ替わっています。

draft 00 では、以下の 2 つの経路のうち、"共用ツール" 経由が先に記載されていました。

- 共用ツール (プラットフォームがあらかじめビルドした OAuth Provider) 経由
- カスタムツール (別テナントが利用している正規 AS を狙って、独自の OAuth Provider を追加) 経由

draft 01 ではこれが逆になり、**カスタム toolkit 経由** の記述が先に、共用 toolkit 経由が後に来る形に整理されています。攻撃者側の目線として、より "能動的" な経路を先に置いたようにも見受けられます。

### 2.2.2 Countermeasures : Connection Context 識別子の要件が精緻化

Document History の "Clarify that COAT countermeasure uniquely identifies each configured AS instance" に対応する変更です。draft 00 では Connection Context 識別子について以下のように記載されていました。

> form a unique connection context identifier, which always includes the unique tool identifier

draft 01 では、これがより厳密に **「Client 側で構成された各 AS インスタンスを一意に識別する」** という要件に置き換わっています。

> form a connection context identifier that uniquely identifies each AS instance configured at the client

すなわち、単に "toolkit 識別子を含める" ことがゴールなのではなく、**Client 内部で構成済みの AS インスタンスを一対一で識別できる形** になっているかどうかが対策の本質であることが明確化された、と読めます。マルチプロバイダやマルチテナント Client 向けの追加要件 (OAuth Provider 識別子 / テナント識別子を含める) の位置付けも、この "AS インスタンスを一意識別する" という上位目的に紐づく形で整理されています。

また、Redirection Endpoint での識別子検証の向きも入れ替わっているようです。draft 00 では以下でした。

> the context identifier from the URI matches with the one in the distinct redirection URI

これが draft 01 では以下のようになっています。

> the context identifier from the distinct redirection URI matches with the one in the user's session

主語 (どちら基準で照合するか) が **redirect_uri 由来の識別子** を基準に、セッション側と一致するかを見る、という向きに整理し直されています。

さらに、Mix-Up 対策による代替の記述についても、脆弱なままとなる Client の条件が更新されています。draft 00 では、「AS メタデータ Discovery を個別エンドポイント定義済みの OAuth Provider を埋めるためだけに使い、**Connection Context 識別子ベースの防御を持たない Client**」が脆弱、とされていました。draft 01 では以下のようになり、記述の焦点が **Issuer Identifier とのカップリング** に移っています。

> solely to pre-populate individual AS endpoint fields, without any coupling with the issuer identifier will remain vulnerable

要は、Discovery で解決した Issuer Identifier を Connection Context の一部として (もしくは実質的にそれに相当する形で) 使わない限り、Discovery を導入したところで守れないよ、というメッセージがより明確になった形です。

### 2.3 Cross-user OAuth Session Fixation : CSRF / PKCE との関係整理

Document History の "Clarify Session Fixation countermeasures and relationship to CSRF and PKCE" に対応する変更です。

draft 01 では、この攻撃が RFC 9700 Section 4.7 の "state をブラウザ (より正確には、フローを開始したユーザーエージェント) セッションと結び付ける" 要件に対する違反として見えることが、より明示的に位置付けられました。

また、**PKCE (RFC 7636) だけでは軽減できない理由** の説明も強化されています。PKCE は "Authorization Request と Access Token Request の暗号学的な紐付け" を担保するものの、この攻撃では **被害者自身がフローの最後まで完了する** ため、その暗号学的紐付けは崩れない (=攻撃者にとって邪魔にならない) というロジックが明記された形です。

用語面では、"OAuth connection" が **"OAuth flow"** に統一されている箇所も複数見受けられます。

### 2.3.2 Countermeasures : 対策の分岐がユーザーセッションの見え方ベースに再構成

2.3.2 も同じく "Clarify Session Fixation countermeasures ..." 起点で書き直されています。

draft 00 では対策の一手目が **「End-User を再認証する」** となっていましたが、draft 01 ではより一般化して **「Client 側の既存セッションを介して、フローを開始した End-User を特定する」** ことが起点として位置付けられました。

> identify the initiating user via their existing session at the client

そのうえで、対策の分岐条件が **Redirection Endpoint でユーザーセッションにアクセスできるかどうか** をベースに再構成されています。

- **Redirection Endpoint でユーザーセッションにアクセス可能** な場合
- **アクセスできない** 場合 (別 Origin / 別ユーザーエージェント配下にある場合など)

さらに、後者のケースの選択肢として、**外部ユーザーエージェント側で Redirection Endpoint から End-User を再認証する** というオプションも新規に追加されています。RFC 8252 の Section 7 のネイティブアプリ向けリダイレクトへの言及もこの流れで整理されています。

### References の更新

参照している他ドラフト / 仕様のバージョンも更新されています。

- **CDFS ドラフト** ("OAuth 2.0 Cross-Device Flows Security BCP"): `-12` (2025 年 9 月 5 日版) → `-15` (2026 年 1 月 23 日版)
- **OAUTH-7523bis**: `-00` → `-05`。著者に F. Skokan が追加され、タイトルも `Updates to OAuth 2.0 JSON Web Token (JWT) Client Authentication and Assertion-Based Authorization Grants` に更新
- **MCP-Spec** への参照は削除

## おわり

draft 01 は Document History にもある通り、新規セクションの追加や大幅な構造変更ではなく、"言葉の整理" と "対策の位置付けの明確化" を中心とした細かい調整の差分でした。

ではまた！
